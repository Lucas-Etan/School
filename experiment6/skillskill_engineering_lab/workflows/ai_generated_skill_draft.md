# 原神兑换码抓取器

从公开页面抓取原神直播码、活动礼包码和日常兑换码；跨多个来源去重；评估有效性/失效状态。

## 适用场景

- 当任务要求你查找并总结当前可用的原神兑换码时。
- 当任务要求你检查某个兑换码是否仍然有效时。
- 当任务要求你从多个来源（直播、日常、活动）聚合兑换码并生成一份报告时。
- 当任务要求你根据页面内容确定兑换码的失效时间时。

## 不适用场景

- 自动向游戏内提交兑换码或代表用户进行兑换。
- 访问需要登录、验证码或仅限App内访问的页面。
- 在没有可验证来源的情况下，生成或编造“可用的”兑换码。
- 购买、出售或交易兑换码。
- 大规模试错猜测兑换码（暴力破解）。

## 输入要求

- 一个或多个指向目标公开页面的 URL 或搜索查询。
- 可接受的来源类型：官方渠道（HoYoLAB、米哈游论坛、哔哩哔哩官方）、半官方（合作游戏媒体）、第三方（游戏攻略站、资讯聚合站）。
- 每个来源必须无需身份验证即可公开访问。

## 工作流程

### 阶段 1 — 来源发现

1. 使用至少三种不同的查询模式搜索当前原神兑换码：
   - 最新版本直播（例如 `原神 X.X 前瞻 兑换码`）
   - 活动/通用码（例如 `原神 兑换码 YYYY年MM月`）
   - 英文来源的国际服（例如 `Genshin Impact [version] livestream codes`）
2. 确认每个来源的类型：官方、半官方或第三方。
3. 在抓取前记录来源名称、URL 和类型。

### 阶段 2 — 抓取

1. 使用 `web_fetch` 以 markdown 或原始 HTML 模式提取页面内容。
2. 对每个页面提取：
   - 发布时间戳
   - 任何内嵌的失效说明（例如“有效至5月11日12:00”、“expires May 11”）
   - 所有匹配兑换码模式的文本片段（12–20位字母数字，通常为大写字母和数字）
   - 若有奖励描述则提取
3. 如果英文来源与同一场直播/活动的中文来源并存，则同时抓取并交叉核对兑换码值。（同一场直播通常会针对不同地区发布不同的代码字符串。）

### 阶段 3 — 兑换码提取与过滤

1. 使用正则表达式提取候选兑换码字符串：`/[A-Z0-9]{6,20}/g` — 但要过滤掉：
   - 版本号（例如 `5.7`、`6.6`）
   - 订单号/UID 模式
   - URL 片段
   - 恰好是大写字母的常见非代码单词
2. 为每个兑换码丰富以下信息：
   - `source_name`、`source_url`、`source_type`
   - `published_at`、`valid_from`、`valid_until`（标准化为 UTC+8）
   - `rewards`（如果有）
   - `evidence_text`（来源中支持该兑换码或失效日期的确切句子）

### 阶段 4 — 去重

1. 按兑换码值（不区分大小写）分组。
2. 对于重复的兑换码，保留元数据最完整的那一条；将所有来源 URL 追加到 `also_from` 字段中。
3. 如果同一兑换码同时出现在官方和第三方来源中，则将置信度标记为 `high`。
4. 如果仅有一个第三方来源，则根据网站声誉将置信度标记为 `medium` 或 `low`。

### 阶段 5 — 失效评估

1. 将 `valid_until` 与当前时间进行比较。
2. 如果没有给出明确的失效时间，则推断如下：
   - 日常兑换码：假设在发布日当天结束失效（23:59 UTC+8）。
   - 直播码：通常从直播结束起 24 小时有效。
   - 发布超过 7 天且没有失效信息的兑换码：标记为 `unknown`。
3. 状态分配：`active_candidate`、`expired`、`unknown`、`unverified`。

### 阶段 6 — 输出

1. 生成便于人类阅读的 Markdown 表格。
2. 生成便于进一步程序化处理的 JSON 结构。
3. 在顶部包含一个“仍可能有效”的摘要部分。
4. 包含关于钓鱼和官方兑换渠道的安全提醒。

## 输出要求

- Markdown 报告：必须包含来源表、带状态的兑换码表、有效候选码高亮部分以及安全提醒。
- JSON：必须包含 `report_metadata`（captured_at、sources_consulted、sources_successful）和一个 `codes` 数组。
- 每条兑换码记录必须至少包含：`code`、`source_name`、`source_url`、`source_type`、`published_at`、`valid_from`、`valid_until`、`status`、`confidence`、`captured_at`。
- 如果抓取所有可访问来源后仍未找到可验证的兑换码，则说明：“在可访问来源中未发现可验证的兑换码。”

## 自检清单

- [ ] 所有来源均可公开访问（无绕过身份验证）。
- [ ] 每次抓取都记录了时间戳。
- [ ] 失效时间已标准化为一致时区（UTC+8）。
- [ ] 未知时间明确标记为 `unknown`，而非猜测。
- [ ] 每条兑换码的置信度均已记录。
- [ ] 在来源表中清晰区分官方、半官方和第三方。
- [ ] 已失效的兑换码与仍为活跃候选的兑换码明确分开。
- [ ] 包含安全警告，避免钓鱼/虚假兑换码。
- [ ] 没有编造的“最新可用兑换码”——每个兑换码都可追溯到来源。
- [ ] 如果某个来源返回 403/被屏蔽，则明确说明；不编造数据。

## 失败处理

- **来源被屏蔽（403/区域限制）：** 记录限制情况，跳过该来源，并在报告中提及。不尝试绕过。
- **页面未发现兑换码：** 检查页面是否需要 JavaScript 渲染。如果需要，注明“页面可能为动态内容，静态抓取不足”。不编造数据。
- **搜索未返回相关结果：** 扩大查询词范围，尝试不同日期范围，或换用其他搜索引擎（如果可用）。
- **缺少失效信息：** 使用合理的默认值（日常兑换码按当日结束，直播推广码按直播结束后 24 小时），并在 `evidence_text` 中标记为 `inferred`。若无法推断，则标记为 `unknown`。
- **不同来源的失效日期相互矛盾：** 使用最早的失效时间作为保守估计；在 `evidence_text` 中注明差异。

## 备注

- HoYoLAB 和米哈游论坛可能对非亚洲 IP 返回 403。如果使用基于云的工具，可能会遇到这种情况 —— 记录失败并继续使用其他替代来源。
- 直播码是分区域的：国际服兑换码（例如 `BacktoSumeru`）与国服兑换码（例如中文短语兑换码）不同。始终确认某个来源覆盖哪个区域。
- 来自第三方聚合站（例如 3DM）的日常兑换码并非米哈游官方发布，而是用户提交或转发的。将其视为较低置信度。
