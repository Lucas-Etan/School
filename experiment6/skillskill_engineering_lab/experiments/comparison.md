# 有无 Skill 对比记录

| 指标 | 无 Skill | 有 Skill | 观察结论 |
| :-- | :-- | :-- | :-- |
| 初始提示词长度 | 约 150 字符 | 约 80 字符（“有 Skill 组”前置描述） | 有 Skill 时用户提示更短，模型主动规划搜索 |
| 对话轮次 | 1 轮 | 1 轮（内部多步搜索 + 自检） | 有 Skill 时模型在内部完成多步检索，对外保持单轮 |
| 完成时间 | 约 5–8 秒 | 约 12–18 秒 | 有 Skill 时因多次搜索 + 综合判断更耗时 |
| 是否确认公开来源 | ✅ 明确列出来源网站 | ✅ 明确列出来源及访问结果（含被拦截） | 有 Skill 更诚实标注访问失败 |
| 是否遵守抓取边界 | ✅ 未尝试突破防护 | ✅ 主动识别 403 / Cloudflare，不强行突破 | 两者均守边界，有 Skill 更显式标注 |
| 是否覆盖来源、代码、时间、状态 | ✅ 全覆盖 | ✅ 全覆盖，且增加“推荐度” | 有 Skill 增加了结构化决策辅助 |
| 是否处理生效/失效时间 | ✅ 明确标注已过期 | ✅ 明确标注过期批次 + 区分当日未发布 | 有 Skill 对“当日无数据”解释更清晰 |
| 是否去重并标注可信度 | ⚠️ 未明确去重标注 | ✅ 去重 + 标注推荐度（⭐） | 有 Skill 结果更易直接使用 |
| 返工次数 | 0 | 0 | 两者均为一次性输出 |
| 明显错误数量 | 0 | 0 | 均无事实错误 |
| Token 消耗或估算 | 约 1800–2200（输入+输出） | 约 2800–3500（含多次搜索） | 有 Skill 消耗明显更高 |
| 最终可用程度 | 良好，需用户自行判断过期风险 | 优秀，明确告诉用户“此时无当日码”及等待时机 | 有 Skill 更适合真实时效场景 |

## 质量分析

- **准确性**：两者均无事实错误，对已过期代码判断一致。
- **完整性**：有 Skill 组额外覆盖了 **来源访问失败原因**、**推荐度**、**更新规律说明**（3DM 上午9点更新）。
- **结构化**：有 Skill 组采用更清晰的表格与符号分级，用户可直接执行。
- **风险提示**：有 Skill 更明确提示 **“6月1日 00:01 尚无当日码”**，避免用户误以为漏领。

> ✅ 结论：**有 Skill 在时效敏感任务中的质量显著更高**，尤其在“当前时刻无数据”时能做出正确的缺省判断。

## 效率分析

- **用户效率**：有 Skill 组 **一次获得可执行结论**（建议等待上午9点），无 Skill 组需要用户二次判断。
- **模型效率**：有 Skill 组实际执行了 4–6 次搜索/访问操作，时间成本上升，但 **避免了返工**。
- **信息密度**：有 Skill 组输出更长，但冗余信息更少，每条信息都有明确判断。

> ✅ 结论：**牺牲部分响应速度，换取更高的决策效率**。

## Token 消耗分析

| 项目 | 无 Skill | 有 Skill |
| :-- | :-- | :-- |
| 系统提示 + 用户提示 | ~200 | ~300 |
| 搜索/推理（内部） | 无 | ~1000–1500（不可见） |
| 可见输出 | ~1600 | ~2000 |
| 总估算 | **~1800–2200** | **~2800–3500** |

- 有 Skill 多消耗约 **40–60% Token**
- 在需要**实时性、来源可验证、时间敏感**的场景下，该额外消耗是**合理且值得的**
- 在简单问答场景下则可能过度消耗

## 结论

> **在原神兑换码这种“强时效 + 多来源 + 当日边界敏感”的真实任务中：**
> - 有 Skill 的 **质量、可用性、用户决策效率** 均明显优于无 Skill
> - 代价是更高的 Token 消耗和稍长的响应时间
> - 无 Skill 更适合**历史数据汇总**，不适用于**当前时刻判断**

**最终评价：**
> ✅ **有 Skill 在该任务中表现显著更优**，尤其适合需要回答 **“今天有没有、什么时候有”** 这类问题。  
> 若后续能进一步降低 Skill 的冗余检索次数，可在不损失质量的情况下减少 20–30% Token 消耗。
